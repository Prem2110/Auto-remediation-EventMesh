Premortem Report

SAP CPI Self-Healing Agent — Sierra Digital
Frame: 6 months from now, the project has failed. Working backward to find every reason why.  |  Generated: 2026-05-26

Synthesis

Most Likely Failure

The async deploy gap (Failure #5). Every fix the agent "applies" may not actually deploy. HTTP 202 is not deployment confirmation — it's a queue acknowledgment. This is already a known open gap (in project memory) and still unresolved. Until post-deploy status polling is implemented, the fix history is untrustworthy and any client audit will expose the discrepancy.

Most Dangerous Failure

The settings page without guardrails (Failure #8). A single misconfigured confidence threshold causes the agent to auto-deploy incorrect fixes to production iFlows, corrupting live business transactions. This is the failure most likely to trigger contract termination and reputational damage — the client has no audit trail and Sierra Digital can't reconstruct what happened.

The Hidden Assumption (across all 8 failures)

"If the pipeline completes without throwing an exception, the fix was applied correctly." This assumption underlies 5 of the 8 failures: the SAP API surface change (the pipeline completed, nothing was actually fixed), the async deploy (202 accepted, deployment failed silently), the silent DB write failure (no exception, status never persisted), the OAuth expiry (update succeeded, deploy failed, no rollback), and the confidence score mismatch (high score, wrong fix). The system has no end-to-end verification that the intended outcome was actually achieved. It only verifies that each step didn't crash.

The Revised Plan — Concrete changes this week

  1. Add post-deploy status polling before claiming success. After deploy-iflow returns 202, poll get-iflow (or get-deploy-status) every 10s for up to 3 minutes. Only mark "Fix Applied" when iFlow reaches STARTED state. On FAILED, call get-deploy-error and rotate strategy. This closes the most likely failure and the hidden assumption at once.
  2. Add a minimum floor and audit log to the settings page. Reject any confidence threshold below 0.75 with a validation error. Log every settings change with timestamp, user, old value, and new value to the incidents database. Show the audit log in the UI. This closes the most dangerous failure.
  3. Add an idempotency lock on incident processing. Add a processing_at / processing_by column to the incidents table. The loop only picks up incidents where this is null or stale (>10 min). This prevents double-processing after HANA outages.
  4. Add a pre-emptive unlock call at fix pipeline start. Call unlock-iflow unconditionally before get-iflow in execute_incident_fix. This was already identified in project memory as a one-liner improvement — it prevents OAuth expiry mid-fix from leaving iFlows permanently locked.
  5. Show scope limitations explicitly in the dashboard. When an incident is Groovy/XSLT/mmap and the system can't fix it, display "Outside remediation scope — manual review required" rather than a generic low-confidence message. Clients need to know this is by design, not a malfunction.

Failure Deep-Dives
Failure #1 High

Fix rate stays too low to be trusted

The Story

The value-search fallback was a patch on a misdiagnosis. The real problem was that LLM RCA produced structurally plausible but semantically wrong fixes for the majority of error types. The +0.10 confidence boost pushed borderline cases (0.82 → 0.92) into auto-fix — but those were the cases that previously routed correctly to manual review. They now deployed in a broken state. The client saw actively broken deployments, not just unresolved alerts. By month 3, they had a rule: flag every agent-touched iFlow for manual review. By month 5, the agent was generating more work, not less. The confidence score measured RCA completeness (all 4 fields present), not fix correctness. A confidently wrong answer scored 0.95.

Underlying Assumption
A high RCA confidence score correlates with a correct fix, when it only measures structural completeness of the LLM's output.
Early Warning Signs
  • Deploy success rate drops while RCA confidence scores trend upward — the two metrics diverge in logs within the first 2 weeks of the boost going live.
  • The same iFlow error type reappears in the incident table on consecutive days — the agent "fixed" it, it was redeployed, and it broke again.
Failure #2 Critical

SAP API surface change breaks pipeline silently

The Story

SAP's Q1 quarterly update moved iFlow XML from the iflowContent key to artifactContent. The _unwrap_getiflow_response function checked only the old key; when absent, it returned the raw JSON blob. The LLM received outer JSON, not XML, and generated patches against phantom nodes. update-iflow accepted the payload. deploy-iflow changed from 202 to 200 with "status":"queued" — the keyword matcher didn't match, but the orchestrator's LLM-evaluation path overrode and marked success anyway. For 3 weeks, every "Fix Applied" record was fiction. A client escalation revealed zero successful deployments since the platform update date.

Underlying Assumption
HTTP status codes and keyword-matched strings in MCP tool responses are a reliable, stable proxy for actual SAP platform behavior.
Early Warning Signs
  • get-iflow output length dropped sharply in logs — <XML len=~400> instead of <XML len=~8000+>. A size threshold alert would fire within the first hour.
  • fix_applied=True with deploy_success=False across consecutive incidents — trivially detectable with a daily SQL query, but no monitoring existed.
Failure #3 High

HANA outage causes double-processing and data corruption

The Story

HANA entered a 45-minute maintenance window during business hours. The autonomous loop hit HDBException: connection refused, retried in a tight loop, exhausted its stack, and crashed. CF's health check (HTTP endpoint only, no HANA dependency) declared the app healthy — the loop stayed dead for 4 hours with no alert. When HANA came back, the loop restarted and found 23 open incidents, including several already manually resolved. With no idempotency key and no processing lock, it re-ran the full fix pipeline on all of them. Three iFlows were redeployed mid-message. Two corrupted in-flight payloads. The client pulled the agent's credentials that afternoon.

Underlying Assumption
HANA Cloud would always be available when the loop ran, so incident state transitions never needed to be idempotent or recoverable.
Early Warning Signs
  • The incident table had no processing_at column — any engineer reading the schema could see concurrent replay was structurally impossible to prevent.
  • Integration test re-runs against a shared dev database occasionally produced duplicate fix attempts, closed as "test harness issues" rather than missing idempotency.
Failure #4 High

System can't fix what clients actually break (Groovy/XSLT)

The Story

In the client's production environment, 65% of iFlow failures originated in Groovy scripts and XSLT transformations — not property/config errors. The system classified these correctly (MAPPING_ERROR, RUNTIME_EXCEPTION), ran full RCA, then stalled at the confidence gate every time with "insufficient confidence to auto-fix." This happened hundreds of times a month. The dashboard showed high incident volume with a 4% resolution rate. The client concluded the agent was malfunctioning — not that it was operating as designed for a problem that didn't match their environment. The limitation was never surfaced in the UI. There was only a low confidence score where clients expected action.

Underlying Assumption
The errors reachable by the fix engine represented the majority of errors the target client actually experienced.
Early Warning Signs
  • During pilot, message log analysis consistently showed Groovy-originated stack traces — the team focused on incidents that resolved, not on those that didn't.
  • Confidence score distribution skewed heavily below threshold from week one. Nobody asked why the median was so low.
Failure #5 Critical

Async deploy creates false "Fix Applied" history

The Story

The agent treated HTTP 202 from deploy-iflow as deployment success. Quota exceeded errors, XML schema violations, and dependency conflicts all returned 202 and then failed asynchronously. The agent closed these incidents as fixed. No polling loop, no status check, no reconciliation. Fixed iFlows reappeared as new incidents with identical error signatures 24–48 hours later — the system re-opened what it had never closed. Six months in, a client compliance audit cross-referenced the agent's fix history against SAP CPI deployment logs. 23% of "Fix Applied" records had no corresponding STARTED state in SAP. The client concluded the agent was fabricating success records. No technical explanation of the 202 async gap recovered the trust.

Underlying Assumption
HTTP 202 from SAP CPI is a reliable proxy for eventual deployment success, rather than merely an acknowledgment that the request was queued.
Early Warning Signs
  • iFlows marked "Fix Applied" reappearing as new incidents within 24–48 hours with identical error signatures.
  • SAP CPI cockpit showing iFlows in persistent STARTING or ERROR state that the agent's database listed as STARTED — visible to any engineer who checked both systems side by side.
Failure #6 Medium

OAuth expiry leaves iFlows locked mid-fix

The Story

During a long overnight session, OAuth tokens (4-hour TTL) expired between update-iflow and deploy-iflow on 3 consecutive fixes. update-iflow had already succeeded — iFlows were checked out and modified. deploy-iflow returned 401. The error handler logged the 401 and moved on without rollback or unlock. The pipeline's cleanup logic only handled "locked" errors on entry, not on exit after partial success. By morning, 3 production iFlows were locked in a modified-but-undeployed state. Subsequent fix attempts hit the locked state and skipped. The client's integration team spent half a day manually unlocking and reverting.

Underlying Assumption
Token refresh failure would only ever occur before a pipeline started, never between an already-committed write and its mandatory follow-on deploy.
Early Warning Signs
  • In staging, deploy-iflow occasionally logged a 401 on first attempt followed by a silent retry — nobody verified whether the retry was fetching a fresh token or reusing the expired one.
  • A small count of "unresolved" incidents after sessions longer than 3.5 hours — treated as noise, not as a locked-state signal.
Failure #7 High

Zero test coverage makes every refactor a gamble

The Story

A HANA schema refactor changed the physical column naming convention. update_incident(incident_id, {"status": "AUTO_FIXED"}) silently skipped the write when the column lookup returned None — the function returned without exception. The logger emitted a warning that nobody was watching. On the next loop iteration, the incident was still FIX_IN_PROGRESS and was picked up again. The loop began re-fixing the same 5 iFlows in parallel across multiple ticks. Within 12 hours, SAP AI Core token consumption triggered rate limiting (429s). All remediation capability went down for 48 hours across the shared tenant — affecting every client. No tests existed for update_incident verifying persistence. The CLAUDE.md 80% coverage standard was never enforced by CI.

Underlying Assumption
Because update_incident never raised an exception, it had successfully written to the database.
Early Warning Signs
  • Structured logs showing update_incident: skipping unknown column 'status' warnings after the schema migration — no alert was wired to this pattern.
  • The FIX_IN_PROGRESS incident count on the dashboard plateaued and grew instead of resolving — status transitions were clearly not completing.
Failure #8 Critical

Settings page is an unguarded kill switch

The Story

A junior IT admin at the client lowered the auto-fix confidence threshold from 0.90 to 0.50, thinking he was tuning sensitivity like a spam filter. No warning, no role check, no confirmation dialog, no audit entry. The system accepted it silently. Within 48 hours, the agent was deploying fixes that previously would have gone to manual review. Weak RCA swapped credential aliases on 3 iFlows. Wrong endpoint URLs were substituted on 2 more (from low-confidence web search results now acceptable under the new threshold). All 5 deployed successfully. Business process failures surfaced 2 days later. The incident response team needed to reconstruct who changed the threshold, when, and which fixes ran under degraded confidence. None of that data existed.

Underlying Assumption
Settings page access would be self-regulating because only technical staff would use it.
Early Warning Signs
  • UAT sign-off never tested what happened when the confidence threshold was set below a safe floor — no validation rule was written and nobody noticed.
  • The first production deploy had no change log for _runtime_cfg values — if anyone had asked "what were the settings when this fix ran?", there was already no answer.
